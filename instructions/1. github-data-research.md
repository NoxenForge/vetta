# 2. GitHub 数据抓取需求分析

> **来源：** PRD v0.1 — GitHub Insight
> **角色：** 数据分析 — 将 PRD 功能需求映射为具体的 GitHub API 数据抓取方案
> **日期：** 2026-07-09

---

## 一、Executive Summary

PRD 定义了 10 个功能模块，核心数据全部来自 GitHub 官方 API。经过逐模块分析，需要抓取的数据分为 **5 大类、9 个 API 端点**，覆盖 Repository 元数据、统计指标、内容解析、搜索发现和社区健康五个维度。

**核心发现：**
- **85% 的数据需求** 可由 3 个端点覆盖：`/repos/{owner}/{repo}`、`/search/repositories`、`/stats/*`
- **最大瓶颈是速率限制**：认证用户 5,000 req/h（核心 API），搜索 30 req/min
- Trending 数据 GitHub API 不直接提供，需要从 `/search/repositories` + 时间窗口查询模拟
- 代码结构分析（目录树、架构）需要 `/contents/` + README 解析，消耗大量 API 配额

## 二、数据需求 → API 映射

### 2.1 按 PRD 模块拆解

#### 模块一：首页 Trending / Latest

| 展示字段 | API 来源 | 端点 |
|----------|----------|------|
| Logo | `owner.avatar_url` | `/repos/{owner}/{repo}` |
| Name | `full_name` | 同上 |
| Description | `description` | 同上 |
| Language | `language` | 同上 |
| Stars | `stargazers_count` | 同上 |
| Growth | 计算：当日 Stars — 前日 Stars（需自行存储快照） | 本地计算 |
| AI Summary | 后续 AI 层生成 | — |
| Score | 自定义算法（Stars + Forks + 活跃度加权） | 本地计算 |

**排序支持：**
- Stars → `sort=stars`
- Growth → 本地计算
- Language → `language:` qualifier
- Created Time → `sort=created` (via search qualifier `created:>YYYY-MM-DD`)

**关键端点：** `GET /search/repositories` — 按 `stars`、`created`、`pushed` 排序，按 `language:` 过滤

#### 模块三：Repository Analysis

| 分析维度 | 所需数据 | API 端点 |
|----------|----------|----------|
| 项目概览 | `description`, `homepage`, `license`, `topics` | `/repos/{owner}/{repo}` |
| 技术栈识别 | `language` + AI 分析 README | `/repos/{owner}/{repo}` + `/readme` |
| 目录结构 | 文件树 + `go.mod` / `package.json` / `Cargo.toml` 等 | `/contents/` (递归) |
| 架构分析 | AI 分析目录结构 + README | `/contents/` + `/readme` |
| 代码导读 | 入口文件、核心模块 | `/contents/` (关键路径) |
| 同类项目 | Search by `topic:` + `language:` | `/search/repositories` |
| AI Q&A | 基于 README + 目录结构 RAG | `/readme` + `/contents/` |

#### 模块四：Metrics Dashboard

| 指标 | 所需数据 | API 端点 |
|------|----------|----------|
| **Activity Score** | Commits + PRs + Issues + Releases 综合 | 多个 stats 端点 |
| **Commit Pulse** | 30天 commits、contributors、additions/deletions | `/stats/commit_activity` + `/stats/code_frequency` |
| **Issue Velocity** | Issues 数量、平均响应时间、平均关闭时间 | `/issues` (state=all, since, per_page) |
| **Release Health** | 最新 Release 时间、版本号、频率 | `/releases` + `/releases/latest` |
| **Community Growth** | Stars/Forks/Watchers 趋势（需自存快照） | `/repos/{owner}/{repo}` 定期抓取 |
| **Tech Health** | License, CI, Security, Dependencies | `/community/profile` + `security_and_analysis` |

#### 模块五：Repository Timeline

| 事件 | API 端点 |
|------|----------|
| 首次 Commit | `/commits` (翻到最后一页) 或 `/stats/contributors` 推算 |
| Star 里程碑 | 需自存 Stars 历史快照 |
| 首次 Release | `/releases` (first page, last item) |
| 重大版本 | `/releases` (全量) |
| 重大 PR | `/pulls?state=closed&sort=popular` |

#### 模块七：GitHub News

| 内容源 | API 端点 |
|--------|----------|
| 仓库 Release Notes | `/repos/{owner}/{repo}/releases` |
| 组织/用户动态 | `/users/{username}/events/public` |

#### 模块八：Widgets / Badge

数据均来自上述端点的聚合计算结果，不产生新的 API 请求。Badge 数据从本地数据库读取。

#### 模块九：搜索

| 搜索维度 | API 端点 |
|----------|----------|
| Repository | `/search/repositories` |
| Organization | `/search/users?type=org` |
| User | `/search/users` |
| Topic | 本地 topic 索引 + `/search/repositories?q=topic:xxx` |
| Language | `/search/repositories?q=language:xxx` |

### 2.2 按 API 端点汇总

| # | 端点 | 频率 | 用途 | 覆盖模块 |
|---|------|------|------|----------|
| 1 | **`GET /search/repositories`** | 每 30 分钟 | Trending、搜索、按语言/话题发现 | 模块一、三、九 |
| 2 | **`GET /repos/{owner}/{repo}`** | 按需 + 每日更新 | 仓库元数据 | 全部模块 |
| 3 | **`GET /repos/{owner}/{repo}/stats/commit_activity`** | 每日 | Commit 活跃度 | 模块四 |
| 4 | **`GET /repos/{owner}/{repo}/stats/code_frequency`** | 每日 | 代码增删量 | 模块四 |
| 5 | **`GET /repos/{owner}/{repo}/stats/contributors`** | 每日 | 贡献者排行 | 模块四 |
| 6 | **`GET /repos/{owner}/{repo}/stats/participation`** | 每周 | 社区参与度 | 模块四 |
| 7 | **`GET /repos/{owner}/{repo}/community/profile`** | 每日 | 社区健康度 | 模块四、八 |
| 8 | **`GET /repos/{owner}/{repo}/readme`** | 按需 | README 内容（AI 分析输入） | 模块三、六 |
| 9 | **`GET /repos/{owner}/{repo}/contents/{path}`** | 按需 | 目录结构（AI 架构分析） | 模块三 |

## 四、数据抓取策略

### 4.1 分层抓取架构

```
┌─────────────────────────────────────────┐
│ Layer 1: 定时批量抓取（高频）             │
│ /search/repositories → Trending 榜单      │
│ 频率：每 30 分钟，每次 100 条 × 5 页     │
│ 配额消耗：5 req / 30min（搜索 API）       │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│ Layer 2: 仓库详情抓取（中频）             │
│ /repos/{owner}/{repo} → 元数据入库       │
│ 对新发现的仓库 + 每日更新已有仓库         │
│ 配额消耗：每仓库 1 req                    │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│ Layer 3: 深度指标抓取（低频）             │
│ /stats/* → Metrics Dashboard 数据        │
│ 每日批量，仅对活跃仓库（Stars > 100）     │
│ 配额消耗：每仓库 4 req                    │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│ Layer 4: 按需内容抓取（用户触发）          │
│ /readme + /contents → AI 分析            │
│ 仅在用户查看 Repository Analysis 时触发   │
│ 配额消耗：每仓库 1-10 req（视目录深度）   │
└─────────────────────────────────────────┘
```

### 4.2 速率限制预算

| API 类别 | 限制 | 预算分配 |
|----------|------|----------|
| Core API | 5,000 req/h | Layer 2+3: ~500 repos × 5 req = 2,500/h |
| Search API | 30 req/min | Layer 1: 5 req/30min（留有 80% 余量） |
| 内容 API | 同 Core | Layer 4: 按需，缓存 README 减少重复请求 |

**结论：** 认证 Token 下的 5,000 req/h 足够支撑每日数千仓库的数据刷新。搜索 30 req/min 的瓶颈可以用缓存策略缓解——Trending 榜单不需要真正实时。

### 4.3 缓存策略

| 数据层 | 缓存时长 | 理由 |
|--------|----------|------|
| Trending 榜单 | 30 min | 搜索 API 限制 |
| 仓库元数据 | 6 hours | Stars 等变化较慢 |
| Stats 指标 | 24 hours | 自身缓存 + 计算开销大 |
| README 内容 | 24 hours | 不频繁变更 |
| 目录结构 | 7 days | 很少变；有 Release 时刷新 |

## 六、风险与注意事项

| 风险 | 等级 | 缓解措施 |
|------|------|----------|
| **速率限制耗尽** | 🔴 高 | 多 Token 轮换；Redis 计数限流；请求合并 |
| **Stats API 202 延迟** | 🟡 中 | 异步队列 + 重试（GitHub 后台计算后返回 200） |
| **大仓库 /contents 递归** | 🟡 中 | 限制深度（max 3 层）；缓存目录树 |
| **Fork 仓库噪音** | 🟢 低 | 搜索时 `fork:true` 过滤；`parent` 字段判重 |
| **语言为 null 的仓库** | 🟢 低 | 通过 topics 和文件扩展名推断 |
| **Trending 非官方 API** | 🔴 高 | 用 `/search/repositories` + `stars:>N` + `created:>date` 模拟；不依赖爬虫 |
| **代码结构分析 API 消耗大** | 🟡 中 | 先用 README 做 AI 分析；只在用户请求时深入 /contents |

---

## 七、建议

1. **MVP 阶段优先抓取 Layer 1 + Layer 2**：搜索 + 元数据即可支撑首页 Trending、搜索、基础详情页
2. **Layer 3（Stats）延迟到第二阶段**：Metrics Dashboard 依赖 Stats 端点，放入下一批
3. **README 分析是 AI 价值核心**：优先缓存 README 内容，这是 AI Summary / Architecture 分析的主要输入
4. **自建 Stars 历史快照表**：GitHub API 不提供 Stars 历史趋势，需要每日抓取 `stargazers_count` 并存入本地时序表来计算 Growth
5. **搜索配额是真正的瓶颈**：30 req/min 意味着每天最多 43,200 次搜索请求，需要精打细算
